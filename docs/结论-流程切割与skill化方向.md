# 结论 · 流程两阶段划分与 skill 化方向

本文记录讨论已收敛的结论与显式待裁点。原理与证据不在此复述：方法论条款见 README 对应章节；superpowers 类外源证据见调研文档；Phase 0 参考的 PM 插件三件外源系对话中直接实读，依据见 §1 内注。

## 1. 两阶段划分（方向已认可；待批子项见 §5）

**划分**：Phase 0 目标明确阶段 / Phase 1 里程碑交付阶段。其实质是给 README 2.C 既有三层文档结构补工序视角——**Phase 0 = 产出并维护「目标层＋节奏权威层（ROADMAP）」的工序；Phase 1 = 逐 Milestone 产出「执行套件层（G-R-D-T）」的工序**。模板已预留接口（Goal 的「阶段目标」、Requirement 的上游均指向目标层大目标＋ROADMAP 行），但此前仓内没有产出上两层的工序，Phase 0 补此空白。

**全景（两阶段构成循环，非单向流水线）**：

```
Phase 0 目标明确（产出并维护：目标层 + ROADMAP）
  想法/问题 → brainstorm（过程层调研稿，免 critic、照常归档）
            → 顶层目标文档（为什么做+损害排序+顶级不变量；critic 审级）
            → ROADMAP（Milestone 划分+各阶段目标+TODO 待分配；critic 审级）
                 │  边界 = 既有立项状态机：登记立项 + 建 G-R-D-T 套件
                 ▼
Phase 1 里程碑交付（每 Milestone 一轮，按 ROADMAP 依赖图可并行）
  Goal → Requirement → Design → Task → 实现
  （文档过 critic，代码过审查，均为既有工序）
  → 关账三纪律 + 关账前核对单 → handoff / 复盘
                 │  关账回填 ROADMAP 状态（同批刷新）；新想法进 TODO
                 └──────→ 回到 Phase 0 维护态
```

**边界与修形结论**：

- 阶段边界不新造门禁：Phase 0→1 的切换动作＝该 Milestone 立项（登记＋建套件，Goal 先行）；Phase 1 出口＝关账三纪律＋核对单＋handoff＋回填。
- Phase 0 分「初建」与「维护」两态：新想法走 TODO→负责人分配→Milestone 内成文（增量轨既有规则）；重排走编号沿革表。
- 技术选型与架构不归 Phase 0：brainstorm 可做可行性假设检验，选型/架构**结论**归选型与架构线；基座是第一个 Milestone 的交付物（均发生在 Phase 1）。Phase 0 排 ROADMAP 时将 M0 排为基座/选型。
- Phase 0 产物同样过质量工序：顶层目标与 ROADMAP 属 critic「审」级；brainstorm 过程稿免 critic、按过程链照常归档（README 2.B）；方向性拍板记决议记录。
- 旁路显式保留：缺陷修复走冻结受控变更闸，单步琐碎改动豁免——不是所有工作都过两阶段全链。
- 不设 Phase 2：发布（上机前核对单）、关账、复盘均为 Phase 1 每轮出口步骤；Milestone 本身即版本级粒度。
- 术语消歧待定名：ROADMAP 模板的「阶段划分」指 Milestone 划分，与 Phase 0/1 的「阶段」撞词，成文时须给两阶段定名并登记消歧（见 §5）。

**Phase 0 参考材料的重定位（推导自既有分池裁决）**：Phase 0 成文环节参考的外源是 Anthropic product-management 插件三件——product-brainstorming／write-spec／roadmap-update（2026-07-18 对话中直接实读原文；未单独成调研文档，本段外源依据即此，不在 superpowers 调研文档覆盖内）。其中 write-spec 产出的标准 PRD 含「需求＋验收标准（P0/P1/P2 逐条带 AC）」段，与「需求按 Milestone 分池、无全集需求池、ROADMAP 先于任何具体需求」相抵。故 Phase 0 的 spec 环节重定位为**顶层目标文档**：吸收 PRD 的问题陈述/目标/非目标/成功度量（目标级），补目标层三问（为什么做＋损害排序＋顶级不变量）；PRD 的需求＋AC 段在 Phase 0 禁用，留给各 Milestone 立项时的 Requirement；Future Considerations 对应 ROADMAP 的 TODO 待分配列表。外源 roadmap 类材料的日历/季度/甘特框架与「不设日历时间盒、判据关账」相抵，吸收其操作（加项/重排/依赖管理）、丢弃日历框架。

## 2. 流程保证机制（skill 按单件切分，顺序不靠阶段大皮）

约束对象主要是 agent；用户「不按流程」的典型形态是从链中间切入。流程的真身在**产物链＋状态机**，不在 skill 包装里。五层机制，从硬到软：

1. **产物门禁（主承重，本仓独有的硬层）**：每个 skill 开头固定「前置产物检查」——本环消费哪份文档、要求什么状态（元信息块状态字段/ROADMAP 状态行/立项状态机）。缺前置不执行本环、显式指回上游 skill。立项状态机即流程令牌，检查项＝文件存在性＋状态字段，可机检。
2. **出口接力**：每个 skill 结尾固定「出口动作」——产出落哪、哪些状态同批刷新、下一环 skill 显式点名（吸收：REQUIRED 接力语法＋终态锁死句式）。流程图分布进节点的前置＋出口，节点怎么切分都不丢。
3. **薄总线路由**：methodology skill 扩展为「状态→下一步」路由（判断项目在链上哪一格，给下一动作），常驻注入且**含上下文压缩后重注入**；按「常驻必须极薄」原则执行——外源规则 <200 词、其实测常驻约 481 词并持续瘦身，本仓以 <200 词量级为目标。单件 skill 的 description 只写「何时用」、不泄步骤。
4. **对抗绕过（修辞层，按需克制使用）**：铁律＋「违反字面即违反精神」公理；借口表条目必须来自逐字记录的真实借口（并入本仓「规则挂事故锚」条款）；Red Flags；按失败模式选写法（明知故犯型用禁令、形状不对型用正面配方、漏填型用 REQUIRED 槽位、条件依赖型用可观察谓词）。
5. **上线压测（skill 的实体验证路径）**：skill 关账前用 subagent 压力测试验证行为合规（3 种以上压力叠加、逼出明确选择、带无引导对照组、重复≥5 次、方差当指标）；压测通过记录才算完成态，文本写完只是纸面态——即关账纪律「完成态声明挂实体验证路径」在 skill 上的应用。

另有天然锚：负责人停点（立项/分配/关账裁决）本身是最硬的流程闸。边界两条：不过度强制（假绿是负资产；重型拦截 hook 不做默认配置，按事故一般化闭环再加）；不追求 100% 不可跳——负责人可豁免，机制目标是「默认走对＋跳步可见」。

## 3. skill 形态结论（方向已认可；模板库转向待批，见 §5）

- **单件切分**（示意名：brainstorm、roadmap、write-goal……；正式定名随 skill 设计单元过「去来源专名」审），不按阶段切大皮；顺序由 §2 机制保证。
- **要求内联，不设独立文档模板文件**：skill 正文给「写作要求＋判据＋自检」，不给填空骨架。拆分文件的判据只有两条——100 行以上重参考、或交给 subagent 原样使用的 prompt 模板（放 references/，对应官方渐进加载的资源层）。
- **契约保形**：被机器检查消费的逐字契约（元信息块五字段 YAML、`R<n>-AC<m>` 编号语法、状态机四词、ROADMAP 定位声明）保留为极短结构契约块内联在 skill 里，不化成散文——这是与外源的真实差异（其文档只被人/agent 消费，本仓文档链要过机检）。
- **不双库**：同一结构只允许一个权威归属。模板库十六件三分归宿——多数蒸馏进对应工序 skill（前置检查/写作要求/最小契约/出口动作四段）；确超百行的少数进 references/；templates/ 不再作为与 skill 并存的拷贝源。「从零启动清单」的第一动作相应从「拷贝模板库」变为「安装 skill」。此项改变 2026-07-16 模板库交付物形态，列 §5 待终审批。

## 4. 外源吸收方案（吸道理不搬形制，去来源专名）

本节为据调研推导、2026-07-18 对话认可方向的吸收方案；条目级取舍随 §5 终审批一并生效。

- **采纳**：① 会话注入含压缩后重注入，注入物＝极薄总线；② description 只写何时用；③ REQUIRED 接力语法＋终态锁死句式，以及「skill 开头设一道检查」的**摆位**（检查的内容形制——状态机令牌与文件存在性——系本仓独有，见 §2 第 1 层，不在采纳之列）；④ 借口表挂真实借口记录（并入既有「规则挂事故锚」）；⑤ 按失败模式选写法四分诊；⑥ skill 上线前 subagent 压测（并入关账纪律实体验证路径）。
- **已有更强、不动**：立项状态机与文档链门禁（外源没有的硬层）；critic 工序（critic 审文本对不对，压测验行为跟不跟，对象不同并存）；派发与回传模板（外源同为 prompt 模板文件路线，互相印证）；「model 参数禁省略」（外源同款警告，双向印证）；「可机检的直接自动化不写成 skill」＝既有测量仪与闸门分界。
- **缓采/不采**：说服学整套框架（其「承诺一致」内核已由编排外化承担，不引入心理学专名）；多平台适配层（无此需求）；「零 slash command」不照抄（保留显式入口无害，自动触发为主）。
- **风险注记**：外源社区最集中批评是过度工程化与 token 开销，其自定 <500 词预算被自家最长 skill 突破。吸收时逐条过「删掉它丢什么实质」，对抗层六件套按失败模式分诊、不全量堆进每个 skill。
- **为何不整体采用外源**：其主链只到单 feature 粒度，无 Roadmap/Milestone 层、无状态机与关账语义；本仓多出产品层节奏权威与硬门禁，外源多出行为级验证与对抗写法，互补拼合即 §2 五层。
