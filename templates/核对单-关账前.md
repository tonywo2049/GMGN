# 模板 · 核对单-关账前

## 适用时机

阶段级关账、契约 / wire 冻结、**推翻既有结论**、对外承诺口径——一切「宣告后回退成本陡增」的状态变更。由主编排者亲自执行；本核对通过是宣告的前置，不是宣告本身。

## 模板正文

**用法**：先对每问自由扫描，再看示例对照——示例是例子不是边界。「项目自填实例区」留给本项目遇到的真实事故往里加。

### 问 1 · 实体验证覆盖——每个关账判据都有实体验证路径吗？

- 逐判据回答：**谁、在哪个真实产物上、跑了什么、看到了什么**？
- **判据源头清单先于证据集**：先枚举判据的全部出处（任务卡完成判据、里程碑行、退出判据、会签条件），逐项对到证据或豁免记录——证据集里没有、又无豁免的判据 = **被静默移出**，扫证据集本身查不出它，必须从源头清单正向对账。
- 哪些判据的证据是**纸面态**（覆盖矩阵、离线折算、推理、格式 / 静态标志）？清单化并显式标注了吗？纸面态不能冒充实体验证。
- "格式确认" ≠ "运行时验证"：承载决策的关键属性，造出触发场景运行过吗？

> 项目自填实例区：
> 泛化示例：① 覆盖矩阵满格（如 37/37）而真实产品端到端路径从未执行过被测逻辑；② 里程碑行明列的两条判据（如长稳态 soak、端到端闭环）在证据集里静默缺席、无豁免记录，只有从源头清单正向对账才抓得出。

### 问 2 · 信任面归零——本域所有输入受理点的「来源」列过了一遍吗？

- **穷举**受理点（受理臂、解析器、导入器、API、反序列化点），逐个回答：它消费的每个输入是"注入值""运行时认证"还是"自权威重算"？
- 有受理点信**注入值**的：有裁决理由留痕吗？没有 = 洞，闭了才能关账。
- 修一处的加固**对称传导**到同类受理点了吗？（只闭一臂、其余臂未对称加固 = 系统病灶还在）

> 工具：[信任面登记表.md](信任面登记表.md)（设计时就该在填，关账时全量刷新核对）。
> 项目自填实例区：
> 泛化示例：① 系统性受理认证缺口——只闭了一个受理臂，其余同类臂未对称加固，成为反复被推翻的病灶；② 穷举时漏列了某个受理臂，成为后续被挖出的洞。

### 问 3 · 正交对抗——至少一道「不吃你框架」的视角跑过了吗？

- 红队是按**红方目标**（怎么双取 / 双花 / 永久冻结 / 凭空增发……按本域定义）、跨文档串机制组织的，还是按你的文档切面组织的？后者只查得出"表述不一致"。
- **实现对照**（声明 vs 真实代码 / 时序，警惕把实现给不出的当既成前提）与**长周期 / 终态**（跨周期、升级、恢复、停摆后状态机仍闭合吗）两道跑了吗？
- 对抗方的**切入锚点**吸收了吗？（即便其结论错，锚点常揭示你的盲区——先假设它看到了你看不到的）
- 自己裁的条款自己（及同源 agent）审 = 作者偏差：本次关账有**外源 / 对抗视角**吗？这是硬工序不是可选。

> 项目自填实例区：
> 泛化示例：① 同源多面评审"意见归零"后，外部视角仍挖出多项高危；② 关账被对抗复核连续数判推翻，每判挖出一个真洞——其中一洞正由"专挑框架外"的透镜挖出。

### 问 4 · 假设显式化——结论依赖的最薄弱假设列出来了吗？

- 哪三个假设塌了、结论就塌？证据强度（实测 / 推理 / 外源背书）逐一标了吗？
- 每个假设**最便宜的证伪动作**是什么？值得现在做吗？
- **推翻既有结论时对称适用**：推翻所需的证据强度不低于建立它；负责人质疑要用"补证据"回应，不是反向跳到另一个未验证结论。

> 工具：[最薄弱假设呈报.md](最薄弱假设呈报.md)（随呈报附上）。
> 项目自填实例区：
> 泛化示例：① 负责人一句话质疑往往命中呈报里未显式化的最薄弱假设，命中率高于内部多轮评审——主动供弹药，别等碰巧扫到；② 单一工具单一视角的"证据"把一个正确结论纠成了错的（推翻的证据强度不足以否证多源观察）。

### 问 5 · 不变量族——本域顶层不变量逐一有负向证据吗？

- 对本域适用的不变量族（如守恒等式、唯一写入口、fail-closed、域绑定、互斥对、授权完整性……按项目定义）**点名核对**：每族的负向测试（该拒的真的拒了）在关账基线版本上绿过吗？
- 兜底核对：每个已知风险**有兜底策略**吗？无兜底的必须落文本闭合；已有兜底的登记残余、不追加机制。

> 项目自填实例区：
> 泛化示例：① 关账门禁把关键负向红队回归（该拒的注入被拒）作为固定转正条件；② 每个已知风险有兜底即可关账，不追求绝对完美（残余登记在案）。

### 问 6 · 文本自洽与知识落账——宣告文本自身核过、后账落了吗？

- **全文反向断言**跑了吗：版本号、术语、交叉引用、残留旧口径，与宣告终态一致？（含冻结索引一致性）
- 本波新事故 / 新教训过"事故一般化闭环"了吗（先问哪个既有问题本应逼出它）？案例进项目教训 memory 了吗？
- 状态刷新：handoff / 决议账 / memory 最新态更新、**历史段下沉**（memory 只留最新一波终态）了吗？

> 项目自填实例区：
> 泛化示例：① 一批传导改动漏改数处、被外源反向断言抓获——此后"全文反向断言"固化为工序；② 新教训入库前先过事故一般化闭环，防止工序退化成越来越长的已知坑清单。

## 填写说明

- 由主编排者亲自过单；六问全绿是宣告的前置，不等于宣告。
- 若项目有整树门禁 / 端到端冒烟脚本，可先跑，但**过脚本 ≠ 本波每个新机制的路径都被逐个验证**——脚本只证"一条路径通"。
- 问 1 与"完成态声明挂实体验证路径""判据源头清单正向对账"（方法论第 3 章关账三纪律）同源；问 3 与"多源正交、作者偏差"（第 5 章）同源；此处只列到时机该自问的问题，纪律全文见方法论正文。
- 反模式：① 从证据集反查判据（漏掉被静默移出的判据）；② 用同源 agent 的"意见归零"替代外源 / 对抗视角。
