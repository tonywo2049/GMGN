# 模板 · 需求文档（Milestone 需求池）

> 标准文件名 `Requirement.md`；本文是所在单元**唯一需求权威**——轨道/切片级文档只能做子集划定、具体化、追踪，不得二次定义需求。

## 适用时机

某个 Milestone 立项时，为本 Milestone 建需求池——**每个 Milestone 一份**，立项时成文。不存在覆盖所有 Milestone 的大需求池（覆盖全面的那份是目标层的大目标）；别的 Milestone 的需求用指针引用、不进本池。

## 模板正文

拷贝围栏内整段骨架，按 `<...>` 逐处替换；固定小标题保留。头部元信息块（YAML 五字段）见《文档元信息块》模板，类型填「需求文档」。

```
# <Milestone 名> 需求池

**定位**：本文是 <Milestone 名> 的需求池——只回答「要什么」，不回答「怎么做」（结构、算法、数据格式留给设计阶段）。上游为 [ROADMAP 的 <Milestone> 行](<指针>) 与 [目标层大目标](<指针>)。每条功能需求配可判定、带唯一编号的验收标准（`R<n>-AC<m>`）；所有数值阈值一律参数化，数值 TBD，本文只钉参数间的约束关系。追踪与验证约定见 §E。

**范围**：<本 Milestone 覆盖的需求面一句话>。<相邻 Milestone / 别的工作线的需求>不在本文，各自另立，需要时以指针引用（不复制条文）。
```

## 填写说明

- **只写「要什么」不写「怎么做」**：需求层钉验收标准，结构 / 算法 / 数据格式一律留给设计。反模式：需求里写死数据结构或算法。
- **每条 AC 必须可判定**：WHEN/THEN 句式，结果可观测、可跑、可查、可枚举。反模式：AC 写「体验良好」「性能达标」这类判不出「过 / 不过」的话——不可判定的 AC 等于没有 AC。
- **负向 AC 不可省**：该拒的输入、不该达成的构造，要显式立一条负向 AC（「WHEN <坏构造> THEN 不可达成」），先证明「不做会错」再建墙。
- **数值一律参数化**：需求层只钉参数间的约束关系（如 `X ≤ Y ≤ Z`），具体数值 TBD、留待验收阶段论证定终值。反模式：把某阈值写成具体数字——上游一改下游全动。
- **一池一 Milestone，别混池**：本池只放本 Milestone 的需求；别的 Milestone 的需求用指针引用，不复制条文（复制必然两处不同步）。
- **编号唯一稳定**：`R<n>` / `R<n>-AC<m>` 一旦分配不复用、不回收；跨里程碑稳定，是追踪矩阵和下游回链的锚。
