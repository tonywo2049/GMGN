# 模板 · 文档元信息块

## 适用时机

新建任何「多步 / 含决策 / 有持久产出」的文档时，头部第一件事就是落这个块。trivial 单步改动（改错别字、加一条规则、重命名）豁免，commit message 留痕即可。判不准时问一句：半年后另一个 agent 要不要靠它理解这件事——要，就落。

## 模板正文

每篇文档以 YAML frontmatter 起手，紧接正文标题：

```yaml
---
目标: <一句话——本文要达成什么，完成/关账判据是什么>
上游: [<来源文档标题>](<相对路径>) — <为什么做这件事，任务/需求/决议从哪来>
下游: [<受影响文档标题>](<相对路径>) — <产出流向谁；产出前可留空，事后回填>
状态: <进行中 | 已关账 | 已归档 | …——示例集非闭集，用项目统一术语（如 草稿-待确认、滚动只增）>
类型: <过程型 | 结论型；有专用模板的填模板名，如 决议记录>
---
```

一篇可挂多个上游、多个下游，各占一行。

## 填写说明

**目标**——一句话写清「做成什么」加「怎么算做完」。判据要可判定（可跑、可查、可枚举），不写「优化体验」这类无法验证的话。

**上游**（必填，最要紧的一格）——可点的相对路径链接，指向**直接触发本文的上一跳**，不是泛泛的顶层目标；逐跳相连即成链，靠它能从任一篇往上回溯到需求根。例外：项目最初几篇的上一跳可能就是顶层目标本身——此时直指目标层即可，链从此起。反模式：上游写不出——说明这篇文档没有存在的锚点，先停下问「这到底是响应哪个需求 / 决议」，别硬造一个。

**下游**——影响到别的文档时回填，产出前可空。上游改了，被指的那篇文档的下游要同步补——单向指针两头一致是纪律。

**状态**——用项目统一的完成术语。「关账 / 已关账」= 完成判据达成，不是「大概做完了」；不可逆的完成宣告须挂实体验证路径，给不出就先停在「进行中」。

**类型**——决定正文用哪套骨架。有专用模板的文档（需求 / 设计 / 任务 / 决议记录 / 交接 handoff / 跨轨登记册 …）直接填模板名，骨架从其模板取（模板库文件自身则填「模板」）；其余通用文档二选一：

- **过程型**（调研 / 测试 / 执行 / 长任务）：正文四段 `目标（要回答的问题·判据） / 方法（路线取舍·环境前置） / 实时过程 / 结果（结论·产出·未决项）`。「实时过程」边做边滚动记录（跑了什么、看到什么、如何裁决），长任务断点续跑全靠它。
- **结论型**（目标 / 需求 / 规划 / 契约 / 决议 / 规格）：正文三段 `定位（答什么·不答什么边界） / 结论条款（带唯一编号） / 依据（回指上游与支撑它的过程文档）`。结论型不重复「实时过程」——过程在对应的过程型文档里，这里只回指。

**通用反模式**：

- YAML 里塞叙述性注释、把一段分析写进字段值——frontmatter 只放结构化指针，叙述归正文；混进去会让机器解析和人读都变脏。
- 字段留着占位符不改（`<来源文档标题>` 原样保留）——等于没填。
- 上游指到顶层大目标而非上一跳——链就断了，回溯时跳不回「为什么现在做这件事」。
