# 模板 · 设计文档

## 适用时机

需求已定、动手写代码之前；一个 Milestone / 一条工作线一份。

## 模板正文

拷贝围栏内整段骨架，按 `<...>` 逐处替换；固定小标题保留。头部元信息块（YAML 五字段）见《文档元信息块》模板，类型填「设计文档」。

```
# <名称>设计——<一句话范围>

## 定位

**本文答什么**：<在哪些已定约束（需求 / 选型 / 契约）之上，给出哪些对象的可落地设计——列清具体交付面>。

**本文不答什么**（边界·非目标，防越权）：

- 不改任何上游文档。对需求 / 契约的修订一律以清单形式列入「接缝与契约面声明」，交由主编排者统一处置，本文不直接改、也不解锁任何冻结门禁。
- 不定 <属于别的工作线的部分>，只在接触面引用其已定接口。
- 不重开 <已定选型 / 已冻结论>，承接定案不翻案。

**贯穿约束**（引项目既定纪律，逐节执行，此处不复述）：<优先级铁律 / 失败即拒 / 数值参数化禁写死 / 每个机制专名溯源到需求编号 …>。
```

## 填写说明

- **定位的「本文不答什么」是防越权的硬边界**：把「属于别的线 / 别的阶段 / 已冻结」的部分显式排除，比只写「要做什么」更能挡住范围蔓延。
- **需求 → 设计对象映射是设计的骨架，不是附录**：先有这张表再展开细节。每个对象缺了需求锚，就停下问「这是响应哪条需求」——答不出即过度设计。反模式：为「将来可能用到」预留的抽象 / 扩展点 / 配置项，无需求锚一律不进设计。
- **接缝与契约面声明只列清单、不动上游文件**：设计文档是施工视图，无权修改被多方会签冻结的契约；想改就走「清单 → 主编排者 → 会签」。反模式：设计里顺手把契约改了，绕过冻结门禁。
- **变更记录是传导账**：每次因上游裁决 / 评审修订本文，记一行（日期 / 改什么 / 依据哪条裁决）——下游靠它核对「已决是否已传导到设计」；只有账本没有正文改动、或改了正文不记账，都是漂移。
- **开放问题给推荐不给定论**：设计者可有倾向，裁决权在上级；「已裁 / 待裁」标清楚，防下游把待裁项当已定去实现。
- **分层展开宁少而准**：每个组件只写「结构 + 关键机制 + 溯源」，实现级细节（具体函数签名、行号）留给代码，不在设计里灌。
