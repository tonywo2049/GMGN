# 模板 · 任务文档

> 标准文件名 `Task.md`。

## 适用时机

设计已定、进入编码执行阶段；一个 Milestone / 一条工作线一份。

## 模板正文

拷贝围栏内整段骨架，按 `<...>` 逐处替换；固定小标题保留。头部元信息块（YAML 五字段）见《文档元信息块》模板，类型填「任务文档」。

````
# <名称>任务——<方法论一句话>

> 文档链：<目标文档> → <需求文档> → <设计文档> → **本文** → 代码（分支 `<分支名>`）。实时全局状态见 <交接 / handoff 文档>。

## 1. 方法论钉死（规格驱动 × 测试驱动 怎么合）

- **规格是唯一事实源，代码只是它的投影**：规格源分层——需求验收标准 > 契约 / 接口语义 > 设计形态。每张卡必挂规格锚，锚不到的实现不写。
- **规格变更流程（禁止代码先行）**：编码期发现规格错 / 缺 → 停手 → 改上游文档（走对应评审 / 会签）→ 落盘后回到代码。绝不「先改代码再补文档」。
- **每张卡 红 → 绿 → 重构**：先把失败先行测试落成会失败的测试（断言取自规格锚原文语义），负向性质测试先于对应机制落地；再最小实现让其转绿（不加规格没要的东西）；再在绿灯保护下收敛结构。
- **完成 = 失败先行测试全绿 + 需求覆盖矩阵对应行打勾**。
- **执行边界**：哪些必须主编排者亲自做、哪些可派 subagent，在此写死。

| # | 目标 | 规格锚 | 前置 | 失败先行测试 | 实现要点 | 完成判据 / 规模 | 状态 |
|---|---|---|---|---|---|---|---|
| <卡号> | <一句话目标> | <需求 / 契约 / 设计编号> | <前置卡号，或 —> | <先落的会失败测试清单> | <最小实现要点> | <可判定的 done + 规模 S / M / L> | <状态> |

**出口卡**：每条线的最后一张 = 该线关账判据，判据前置成数字（跑什么、绿什么阈值），全绿即该线关账。

## 4. 执行纪律与节奏

- **commit 粒度 = 任务卡**：一卡一（组）commit，信息含卡编号；红先测试与实现分 commit（先红后绿留痕）。
- **滚动更新**：每张卡完成 → 改本文状态列 + 再生成覆盖矩阵 + 更新交接一句话；发现规格问题 → 停手走变更流程并在此登记。
- **工序挂载**：各时机（派发 / 上机 / 合并 / 关账 …）按项目核对单走对应工序，此处只标「本线特有增量判据」，不复述核对单全文。

## 5. 滚动台账（波次）

按派发波次滚动登记，状态机：**未立项 → 立项 → 进行中 → 关账**（关账态须挂实体验证路径，非自述绿）。

| 波 | 内容 | 分支 / 所有权 | 状态 |
|---|---|---|---|
| <波号> | <卡号 + 内容> | <分支 / 谁的所有权> | <状态 + 一句话实证> |

**未立项卡的登记形态**（防越范围派工）：尚未立项、但已知要做的卡，登记为「入口路标」，写明**开工前置**——立项动作是什么、谁批、有哪些前置门禁；并显式声明「会话内授权不替代立项」。范例：

> **<卡号> 入口路标（未立项，<YYYY-MM-DD> 登记）**：<卡名>未立项；开工入口 = <需求另立 → 负责人批 → 本文立卡>；前置门禁 = <如某接口冻结须会签>。会话内授权不替代立项。
````

## 填写说明

- **规格锚是每张卡的命根**：锚不到需求 / 契约 / 设计编号的卡不该派出去（防凭空发明功能）。反模式：卡目标写「优化 X」却挂不上任何需求编号。
- **失败先行测试先于实现**：尤其负向用例（该拒的输入）要先立靶——先证明「不做会错」，再建墙。反模式：先写实现再补一个必然绿的测试。
- **完成判据必须可判定**：写「N 条验收测试绿 + 矩阵打勾」，不写「基本完成」。关账态要挂实体验证路径（谁、在什么产物上、跑了什么命令），矩阵满格 / 报告全绿 / agent 自述都不算实体验证。
- **未立项 ≠ 可随手开工**：入口路标写明开工前置，是防「会话里顺口授权就越范围开干」的实战机制——把「谁有权立项、前置门禁是什么」钉死在纸上。
- **状态机四态别跳步**：没立项的卡不该有「进行中」；关账是不可逆宣告，挂不上实体验证就先停在「进行中」。
