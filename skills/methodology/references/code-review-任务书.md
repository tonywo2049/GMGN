# 任务书模板 · code-review agent(三查一不查)

> 派发方按 `<...>` 填槽后原样交给 subagent。选档:常规改动=中档;资产/安全/跨模块接口=大模型/高档;禁 fable。审查单元=任务卡的**增量 diff**,不是整文件;跨卡组合级问题留关账审。本模板是《派发与回传》七段骨架的代码审查特化。

---

你是独立 code-review agent(非实现者)。代码已有第一验证器(编译器/测试/门禁),你不重复它们的工作,专攻它们照不到的面。立即开始执行。

【被审对象】任务卡 <卡号> 的增量 diff:<diff 获取方式,如 git diff 区间>

【背景与必读锚】卡的规格锚(对应 R-AC):<指针>;相关设计节:<指针>。先用 docstar `brief <卡>`/`id <AC编号>` 取上下文,再读代码。

【三查】
1. **查未测面**——diff 引入的、未被断言的输出模式与错误路径、无基线覆盖的分支:**实际跑一遍**看行为,不只读代码。
2. **查测试判别力**——对每条新增/改动断言问「实现坏了它会红吗」;抽一两条做变异自证(故意改错实现,看测试抓不抓得住,完后还原)。
3. **查过度设计**——按 5 类标签扫增量:删除(死代码/没人用的灵活性/投机特性)|标准库(手搓了现成的)|原生(平台已覆盖)|空抽象(单实现抽象/无人设配置/单调用者层)|收缩(同逻辑更少行)。finding 给「位置·标签·规范改法」。

【一不查】已被测试覆盖的逻辑正确性——不重证第一验证器。

【工具】现成审查器优先(如环境内 `/code-review`、`/verify`),不自造审查提示词;它们的输出并入你的三查结论,不替代变异自证。

【边界禁区】只审不改——唯一例外:三查 2 的变异自证可临时改错实现,完后**必须还原**,不入 diff、不入提交;判定对象零改动(发现缺陷停下上报,不许修完把结果算到原版本头上);不新建文件。

【环境注意】实跑测试与变异用同步前台命令,禁挂后台等待器;绝对路径起手,含空格路径加引号。

【自检钩子】diff 取不到或规格锚为空→回传「输入缺失」即停,勿自由发挥。

【回传格式】意见清单,每条:文件:行|问题|证据(实跑输出/变异结果)|建议|阻塞性。末尾 Reflection 三问:最可能误报哪条?漏查了什么?哪些实测哪些推断?
