# 文档写作契约(全链共享;各 write-* skill 只内联本文档特有项,通用项以本文为唯一叙述点)

**适用对象**:流程产出的链上文档(白皮书/ROADMAP/G-R-D-T/调研/决议等)。skill 文件与任务书模板等**运行件不适用**(其元数据走 skill frontmatter);使用项目可在 docstar conventions 用 `archive_globs` 把运行件排除出语料。

## 1. 元信息块(每份文档头部 YAML)

```yaml
---
目标: <一句话:本文回答什么>
上游: [<显示名>](<相对路径,真实链接>)
下游: [<显示名>](<相对路径>)   # 尚不存在则写占位说明,建成后同批补链
状态: <草稿 | 已定稿-待批 | 已批准 | 已关账>
类型: <白皮书 | ROADMAP | 目标文档 | 需求文档 | 设计文档 | 任务文档 | 调研 | 决议 | 复盘 | handoff>
性质: <规范 | 记述>   # 固定 token,出生即标:规范=下游依赖其条款,变更须传导;记述=记录,不设门禁
---
```

上游/下游必须是**真实 markdown 链接**(方向键对,docstar 做互惠检查:两端都要认账),不写纯文字。

三条注:① **批准主体分层**(负责人裁决 2026-07-18)——白皮书与 ROADMAP=critic 后负责人批准;G-R-D-T 链内文档=critic 后编排者审核(状态填「已批准」即指过了对应层级);关账=负责人。② **链根文档**(如白皮书)无上游可填,写「无(链根)」。③ 本节收编 templates/文档元信息块.md;两处枚举的唯一权威归属待负责人终审批,差异期以本节为 skill 链适用版。

## 2. docstar 写作四约定

1. **实体放类型化小节、粗体命名**:需求/验收标准/参数/任务写在 `## 需求`、`## 验收标准`、`## 参数`、`## 任务` 类标题下,每条 `- **名或编号** — 说明`。粗体文本即实体身份;小节外的普通粗体不入图。
2. **关系写成链接,不写散文**:依赖关系放 frontmatter 键或行内 `[链接](x.md)`;只在句子里提到的依赖不成边。引用他文某节写 `<文档名> §N`。
3. **术语在定义处标注**:自造术语首次定义放 `## 术语表` 小节(`- **词**: 定义`),使「用了没定义」可检。注意:docstar 默认词表只认「术语表/名词解释/词汇表/glossary」,裸「术语」不识别。
4. **性质出生即标**(见上表)。

## 3. 先图后文(读的纪律)

要了解文档链,**先跑 docstar 再 Read/grep**:全局链 `graph`;单文档进出边 `doc <名>`;编号何处定义引用 `id <ID>`;任务闭包 `brief <task> --json`(派发 coding agent 时把结果贴进任务书)。改完文档、交接前跑 `verify`(改动没打断引用);关账门禁跑 `check --gate <键清单>`——键名=该仓 `check --json` 顶层键,由使用项目钉死;**裸 `--gate` 不设防,有缺陷也退 0**。

## 4. 状态机与同批刷新

状态机四词:`未立项 → 立项 → 进行中 → 关账`(适用于 Milestone 行/切片/任务卡;文档自身状态用 §1 枚举)。任何确认/变更事件发生时,登记该状态的**所有表征处同一批刷新**(如卡关账→台账+追踪矩阵+ROADMAP 行);做不到同批的,唯一合法出口是在滞后处显式标「此处滞后于上游」。

## 5. 使用项目落地

使用项目在仓根放 `.docstar/conventions/conventions.json`(可从 DocStar 自带 `fixtures/methodology/` 预设起步),收编 `R<n>-AC<m>` 等编号语法与任务表列角色(spec/prereq/red/status)。下游暂缺时的 frontmatter 占位用 conventions 声明的 `nonlink_prefixes` 前缀(如「待建:」),免落断链检查桶。
