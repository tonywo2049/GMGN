# 模板 · 核对单-合并前

## 适用时机

合并代码分支、采纳 subagent 产物、采纳外部报告 / 外源产物入权威文档、契约或文档传导落盘——一切「并行产出 → 共享基线」的汇入动作。由主编排者亲自执行，逐问回答留痕。

## 模板正文

**用法**：先对每问自由扫描（把问题套到你的具体场景上），再看示例对照——示例是帮助识别的例子，**不是范围边界**；「项目自填实例区」留给本项目的真实事故。

### 问 1 · 集成完整性——这个变更的实际影响面比它的文件 diff 大吗？

- 共享接口（函数签名 / arity、wire 结构、schema、接缝文件、共用常量）动了吗？**所有消费方**在合并结果里重新验证过吗？
- 验证跑的是**合并后的组合**，还是各卡自己的工作树？（文件级零碰 ≠ 集成无缝——单卡自测看不到跨卡的缝）
- 语义级冲突（两卡各自正确，合起来引入新路径 / 新组合）扫过吗？

> 示例（泛化，非范围边界；项目真实事故往此加）：① 一卡改了消息 envelope 的 wire 结构，与并行卡的旧 arity 冲突，文件级零碰、整树却编译失败；② A 卡加退款、B 卡加重放，各自都对，合起来同一笔被取两次（组合落在按文档分派的缝里）。

### 问 2 · 验证强度不降——合并让任何既有验证变弱了吗？

- 被删 / 被改的**断言、测试、门禁、阈值**，每处有裁决理由留痕吗？（删断言可能合法——重构、换权威锚——但无理由不得过）
- 新增的 ignore / skip / 超时放宽，有**转正条件**和去处吗？
- 夹具 / mock 的适配改动，是"硬化断言"还是悄悄放宽了判定？

> 示例（泛化，非范围边界；项目真实事故往此加）：① 把"断言零删改"设为每次合并绿灯的固定条款；② 夹具适配时顺手放宽了一个断言，判定实际变松却没人裁。

### 问 3 · 声明-磁盘对齐——所有「已完成」声明，磁盘证实了吗？

- agent 自述的分支 / commit / 绿灯 / 自愈，**亲核父链与内容**了吗？（completed ≠ 做完；"我修好了" ≠ 修好了）
- 你有没有把**返回值、通知、文件时间戳**当成了事实？（只有磁盘内容是事实）
- 报告里的数字，是**运行实测**还是**折算 / 预算 / 推理**？形态相同、效力天差地别——引用时标注了吗？

> 示例（泛化，非范围边界；项目真实事故往此加）：① agent 报"待命 / 已完成"实为空转死态，逐分支核后其中一个仍需返工；② 一个离线折算的比率被后续文档当成实测数据反复引用。

### 问 4 · 操作-形态匹配——你要用的工具操作，在当前数据形态下的退化行为清楚吗？

- 逐单元操作（rebase、逐行替换、逐文件迁移）遇到**退化单元**（空变更、撞名、重复、超长）时会怎样？
- 批量操作**中断在半途**的残留形态（合并中止标记、detached HEAD、index 锁、半写文件）如何识别与恢复？
- 工具报"成功"但结果为空 / 部分的场景（编辑幻觉成功、heredoc 污染长文档）排除了吗？

> 示例（泛化，非范围边界；项目真实事故往此加）：① 只含进度更新的 commit 被逐个 rebase 取 theirs 后变空 commit、落进 detached HEAD；② 用 heredoc 写长文档时编码污染了目标文件——工具异常即停、以版本库为权威。

### 问 5 · 命名与数字溯源——汇入的每个机制专名 / 关键数字，能溯到定义它的裁决 / 规格吗？

- 新出现的**机制专名**：溯源到裁决编号了吗？溯不到即幽灵、即删。
- 关键**数字 / 标签**：定义行在哪？派生数字携带假设链（由什么前提推出、前提是否仍成立）吗？
- 说明性 / 论证性文字落盘后**会被读成承诺**：负面裁决的出路说明保持负面语态了吗？

> 示例（泛化，非范围边界；项目真实事故往此加）：① 一段论证性文字的语态漂移，被下游读成了"要做某功能"的机制承诺（幽灵需求）；② 某任务的临时前提数字被贴上权威标签、跨文档传播成硬指标。

### 问 6 · 过度设计扫——这段 diff 能更少、更直接吗？（结构质量轴；代码卡合并必扫，文档同源）

- 按 **5 类标签**逐扫：**删除**（死代码 / 没人用的灵活性 / 投机特性）｜**标准库**（手搓了标准库或框架已有的）｜**原生**（做平台或语言原生已覆盖的事）｜**空抽象**（只一个实现的抽象 / 没人设的配置项 / 只一个调用者的层）｜**收缩**（同一逻辑可更少行）。5 类是扫描辅助、非互斥划分，相邻两可归取其一即可。
- finding 给可判定格式：`位置 · 标签 · 要删的具体构造 · 规范替代`——不写"感觉有点复杂"这类不可判的话。
- **只审这一轴**：正确性 bug / 安全洞 / 性能各走专轴（红审 / 信任面登记表 / 性能门禁），别在本问混判——单轴纯净才不互相稀释（多源正交的应用）。
- 有意保留的简化（为可读 / 为兜底而不最优）就地标注"有意简化 + 理由"、事后统一收割，不让 reviewer 反复提同一处。

> 示例（泛化，非范围边界；项目真实事故往此加）：① 一个只有单一调用者的抽象层被引入"以备将来扩展"——按空抽象内联掉；② 手搓了一段标准库已有等价的解析——按标准库替换。
> 机检：过度设计可用自动化工具作测量仪列 findings，但**是否删由主编排者裁决**（测量仪 ≠ 闸门）。

## 填写说明

- 由主编排者亲自过单——合并、门禁是不可外包给执行 agent 的动作。
- 溯源方法：grep 机制专名 / 关键数字 → 定位定义它的裁决编号或规格行（人工核，溯不到即幽灵、即删）。引用链工具（若项目有）成熟后可接入为机检辅助，但**过工具 ≠ 本问完成**。
- 各问示例区随事故一般化闭环增长：新事故先问"哪个既有问题本应逼出它"，有就往那一问加实例，没有才提请新增问题。
- 反模式：① 只看文件 diff 判影响面，漏掉共享接口的跨消费方波及；② 把 agent 的"已完成"当磁盘事实，不亲核父链。
