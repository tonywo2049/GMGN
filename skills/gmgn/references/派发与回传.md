# 模板 · 派发与回传

## 适用时机

主编排者每派出一个 subagent（编码 / 调研 / 评审 / 执行 / critic）时，用本模板组装任务书；agent 回传时按本模板的回传格式。任务书是 agent 唯一稳定拿到的上下文——它必须自包含。

## 模板正文

### A. 任务书骨架（派发时逐段填，不整段省略）

1. **背景与必读锚**：一句话说清这活从哪来、为什么做——agent 没有主编排者的上下文，任务书必须自包含；随后列**必读锚**（开工前必须读的文件清单：路径 + 各自读什么）。
2. **目标与关账判据**：完成 = 什么可验证状态；能机检的判据直接给命令。
3. **授权范围与禁区**：能动哪些文件 / 目录；硬性禁区点名（测试夹具、他人任务的所有权文件、并行工作树目录等）。
4. **须遵守的工序**：按本项目的时机-工序触发器表，为本任务选取对应工序并点名引用。示例（非穷举）：上真实环境前 → [核对单-上机前.md](核对单-上机前.md)；汇入共享基线前 → [核对单-合并前.md](核对单-合并前.md)；不可逆宣告前 → [核对单-关账前.md](核对单-关账前.md)；新增消费外部输入的入口 → [信任面登记表.md](信任面登记表.md)；产出实质性文档 → 独立 critic（先按范围分级判触发，方法论 §5.1）；改代码、commit 前 → 代码审查（方法论 §5.2，独立于 critic）。微任务无一命中时写「无适用」即可，不硬造。
5. **环境注意**（按任务相关性带）：涉 shell / 文件操作的任务默认全带——绝对路径起手（多工作树下当前目录会漂移）、就地编辑用对平台参数、含空格或非 ASCII 的路径加引号、长命令一律同步前台执行且**禁挂后台等待器**（等待器易陷"进程活着但永不返回"的死态）；纯读 / 纯文本任务只带适用项（至少保留"禁挂后台等待器"）。
6. **自检钩子**：任务书读不到 / 为空 → 回传"任务书缺失"即停，勿自由发挥。
7. **回传格式**：写明"三段必齐，缺段打回"（见下）。

### B. 回传格式（三段必齐，缺段 = 打回重做）

1. **成果与实证**：每项完成声明附实证（命令 + 输出摘要 / commit hash / file:line）——"已完成"式空报告不作数。
2. **偏离与待裁**：偏离任务书的每个决定及理由；任务书未覆盖的歧义列为待裁项，不自由发挥。
3. **Reflection**（轻量三问，逐问作答）：
   - 本次结论依赖的**最弱假设**是什么？
   - 有没有被我忽略的**反例或替代解释**？
   - 我声明的事实哪些是**实测**、哪些是**推断**？（逐项标注）

特化：critic 类任务的回传主体与 Reflection 以独立 critic 工序（方法论 §5.1）为准（意见清单 + 误报 / 漏查 / 实测三问），覆盖本节通用格式。

## 填写说明

### 选档（第 3 步之前先定 model / 档位）

按任务复杂度定档，选定的档及理由写进派发时的"拆解与并行度判断"外化输出：

| 档 | model / effort（能硬控时生效，见下方边界注） | 适用 |
|---|---|---|
| 机械执行 | 小模型 / 低档 | 批量替换、格式核对、清单打印、跑脚本收输出——判断密度≈0、失败成本低 |
| 常规执行 | 中模型 / 中档 | 按明确任务书的单线编码、单文件调研、常规检索汇总——有判断但路径清晰 |
| 判断密集 | 大模型 / 高档（及更高档） | 设计与裁决支撑、critic / 红队 / 对抗复核、归因调查、多文件或触接缝的编码、进入裁决 / 契约 / 规格 / 关账依据的结论产出 |
| 禁派档（可选，项目自定） | 仅供主编排者、不派 subagent | 若项目约定某档专供主编排者，则它不作为 subagent 的派发目标——原因见下方"续跑模型漂移" |

档位名沿用 GMGN 第 4 章的小 / 中 / 大模型说法；落到具体平台时再实例化。

三条规则：①按本机派发工具实际支持的字段显式指定 model / effort；不支持的字段不伪造,改由预定义角色和任务书约束；②选档及理由写进派发前的外化拆解清单（哪些并行 / 派几个 / 哪些自己亲做 / 哪些挂起）；③拿不准取高一档（低档返工成本 > 高档 token 成本）。

边界注：不同派发通道支持的粒度不同——例如有的 per-call 只能定 model 无 effort，有的 model + effort 都能定，有的靠预定义 agent 类型 frontmatter 硬挂 effort。**以本机工具 schema 实测为准，公开文档与本机不符时以本机为准**。无硬控通道时，靠任务书文字引导推理深度。

### 主编排者验收规则（回传格式之外的净增）

- **打回方式**：点名缺什么，要求补齐后再采纳。
- **agent 自述不作数**：涉及磁盘状态的声明（commit / 分支 / 文件）一律亲核；身份与模型只在平台提供可核验运行记录时核验,否则不作无法验证的合规声明。
- **续跑模型漂移**：恢复 / 唤醒 / 续跑能否指定 model 取决于平台,先查本机工具 schema。无法硬控时按**剩余工作性质 × 已积累进度**权衡：机械收尾可续跑并加强验收；判断密集核心段优先新派合适角色；中间产物已落盘时把指针交给新 agent 接续。
- **completed ≠ 做完**：回传呈"待命 / 等通知"等非成果形态，按"进程活着但永不返回"的死态处置（唤醒并明令同步执行）。

**反模式**：① 不查本机工具能力却写死平台参数或模型名；② 拿 agent 一句"已完成 / 已修好"当验收依据而不核磁盘——completed ≠ 做完。
