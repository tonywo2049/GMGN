# 模板 · 分配账本（存量映射账，可选）

## 适用时机

**仅当**项目在采用本体系之前已积累了跨 Milestone 的需求存量（一批早于阶段划分就已成文、带编号的需求 / 验收标准）时才用——把存量一次性划入新 Milestone、划完归档。**没有历史存量的新项目不设它**（定位与边界详见下方骨架与填写说明）。

## 模板正文

拷贝围栏内整段骨架，按 `<...>` 逐处替换；固定小标题保留。头部元信息块（YAML 五字段）见《文档元信息块》模板，类型填「分配账本」。

```
# <存量基线名> 分配账本（<N> 条存量 → Milestone 归属）

**定位**：本文是**存量 <N> 条 → 新 Milestone 划分账**——把 <存量基线文档> 这批早于阶段划分的存量需求 / 验收标准，逐条分配到某个 Milestone（**施工归属**：哪个 Milestone 把它做出来并验收，非「在哪定义」）。**边界（重要）**：**未来新需求不再经本账**——走 [ROADMAP 的 TODO](<指针>) 登记 → 负责人分配进某 Milestone → 该 Milestone **自建需求池**成文；本账只管这批**存量 <N> 条**，随对应 Milestone 过程受控演进，划定后归档。

> **两层分明**（不混）：本账管 **存量条目 → Milestone**（这条归哪个里程碑交）；各套件追踪矩阵管 **条目 → 测试**（这条哪个测试盖）。

| 编号 | 一句话要旨 | 建议归属 | 置信 |
|---|---|---|---|
| <R1-AC1> | <一句话> | <Milestone（切片）> | <高 / 低·待复核> |
| <R1-AC2> | <...> | <...> | <...> |
| … | … | … | … |

## 表尾统计（按 Milestone；切片小计括注）

| Milestone | 条数 | 占比 | 切片构成（历史标签可保留） |
|---|---|---|---|
| **<M1>** | <n1> | <%> | <切片:数 / 切片:数> |
| **<M2>** | 0 | — | <为何为 0 见下说明位> |
| **合计** | **<N>** | 100% | — |

**核对**：<各 Milestone 小计求和> = <N>（= 存量总数 = 逐行条数）；小计、逐行两处必须对账，对不上即有条目漏分或重分。

**某些 Milestone 为何为 0**：<逐一说明——如某 Milestone 的验收方式不产生本层条目、或某增量 Milestone 属全新需求面不在本存量覆盖内；不能留空，否则读者无法判断是真没有还是漏了>。

## 归档声明

存量全部划定、负责人逐条复核低置信项完毕后，本账**一次性处置完成**，状态改「已归档」；此后新需求走 ROADMAP TODO，不回本账追加。
```

## 填写说明

- **它是存量处置，不是长期机制**：只处理「早于本体系」的那批存量，一次划完即归档。反模式：把它当常设的「需求 → Milestone」分配中心，新需求也往里塞——新需求的正道是 ROADMAP TODO → 负责人分配 → Milestone 内成文。
- **归属列格式 = Milestone（切片）**：Milestone 级是分配结论，括注的切片名是施工单元（可保留历史标签）。归属回答的是「谁把它做出来并验收」，不是「它在哪定义」。
- **小计必须与逐行对账**：表尾各 Milestone 小计求和 = 存量总数 = 逐行条数，三者对不上就是漏分 / 重分。反模式：改了逐行归属却不刷表尾小计——账本自身先漂移。
- **低置信项显式标注、留待负责人复核**：拿不准的归属标「低·待复核」，转正 ≠ 转高置信；负责人逐条批后才去标。别把「暂定」写成「已定」。
- **0 条的 Milestone 要给理由**：某 Milestone 小计为 0 不能留空，须说明「为何本层无它的条目」。
