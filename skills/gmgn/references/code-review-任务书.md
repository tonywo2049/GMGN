# 代码审查工序 · 原生入口 + 统一补查

> run-task 收卡与 close-milestone 组合审的执行说明。审查方必须独立于实现者,对象限定为任务卡或里程碑的增量 diff;只报 finding,不在审查中修代码。跨卡组合级问题留到关账审。

## 1. 选择当前平台入口

- **Claude Code**:默认派独立只读 reviewer 使用本任务书。只有“一张任务卡=一个 GitHub PR”且负责人已授权读取 PR、发布 GitHub 评论时,才可用官方 `/code-review`;它不是任意本地 diff 审查器,也不补一般测试覆盖。运行验证用项目命令,`/verify` 可用时可辅助。
- **Codex**:交互用 `/review`;CLI 按对象选 `codex review --uncommitted`、`codex review --commit <sha>` 或 `codex review --base <branch>`。需要 GMGN 专用标准时改用 `codex review '<审查范围 + 本工序要求>'`;范围 flags 与自定义 prompt 互斥。项目测试和真实路径仍要单独运行。
- **其他环境或原生入口不可用**:派独立只读 reviewer,使用下方统一审查任务书。不得因平台缺少 slash command 而取消审查。

平台原生入口只是执行器,不替代本工序的范围、独立性和证据要求。模型、推理档位、参数与入口能力以本机工具 schema 和帮助为准;共享契约不承诺特定模型家族、扇出方式、自动读项目规则、自动跑完整测试或发现全部缺陷。审查回合不主动应用修复,但可能运行局部检查并留下缓存等副产物;结束后执行 `git status --short`,未知新增物先上报,不擅自删除。

## 2. 三查一不查 + 规格锚

1. **未测面**:找 diff 新增但未断言的输出、错误路径、无基线分支和同类调用路径。Bug 修复若只覆盖题面路径、没覆盖共享根因的相邻调用者,上报。
2. **测试判别力**:逐条问新增/改动断言「实现坏了它会红吗」;抽一两条做变异自证,但临时改错只能发生在隔离副本,不得触碰被审工作树。只读环境无法建隔离副本时,要求 coder 提供可重放的变异证据,reviewer 只核命令、输出与还原状态;证据缺失就记未验证缺口。
3. **过度设计**:只审复杂度,用冻结闭集五类——删除、标准库、原生、空抽象、收缩。过度设计指任务不需要的结构,不是“类太多”或“代码太长”;测试、边界保护、安全和明确需求不因减行数而删。
4. **不查**:已经由判别性测试覆盖的逻辑正确性,不重复第一验证器。
5. **规格锚对齐**:把 diff 行为逐条对回任务卡的 R-AC;超出规格是越范围,有 AC 无实现或测试是缺口。

复杂度 finding 统一写成: `位置 · 标签 · 要删的具体构造 · 规范替代`。正确性、安全、性能 finding 走各自轴,不得伪装成复杂度意见。

## 3. 独立 reviewer 任务书

你是本任务卡的独立代码审查 agent,不是实现者。立即开始,只查不改。

【被审对象】任务卡 `<卡号>` 的增量 diff:`<diff 获取方式>`;规格锚:`<R-AC 编号与指针>`;项目规则:`<AGENTS.md 或等价规则文件>`。先取规格原文,再读 diff 和必要调用链。

【检查】按上方「三查一不查 + 规格锚」逐项执行。优先重放实现者的测试与真实路径清单;仅在结果不一致、命令缺失或覆盖不足时扩查。测试判别力变异只在隔离副本执行;只读环境无法隔离时核验 coder 的可重放证据。结束前用 `git status --short` 确认被审工作树未被变异污染。

【边界】只查不改;判定对象零改动;不新建文件。diff、规格锚或项目规则缺失时回传「输入缺失」并停止,不得自由发挥。

【回传】按严重度列 findings,每条给位置、证据、影响和规范改法;复杂度项用固定格式。无 finding 时说明实际覆盖范围。最后 Reflection:最弱假设?忽略了什么反例?哪些实测、哪些推断?
