# 模板 · 核对单-上机前

## 适用时机

判定、压测、soak（长时间持续运行）、长任务战役——一切「运行结果会被用来做决策」的活动开工前。跑之前逐问回答，答案写进该活动过程文档的「方法」节；答不出的问题，就是最可能出事的面，先补齐再开跑。

## 模板正文

**用法**：先对每问自由扫描（把问题套到你的具体场景上），再看示例对照——示例是帮助识别的例子，**不是范围边界**；「项目自填实例区」留给本项目的真实事故。

### 问 0 · 这次运行要回答什么问题？

答案的判据先于运行写下：什么结果算"答完"、什么信号算"判死"，答完即停。一个 run 只答一问；想答第二问，另立 run。

> 示例（泛化，非范围边界；项目真实事故往此加）：run 没有主问题就开跑 → 空烧机时、跑完不知道验证了什么。

### 问 1 · 环境实体性——你依赖的每个实体，真实状态与你以为的一致吗？

- 文件、数据、二进制、依赖库：**存在吗？是正确版本吗？真的物化在磁盘上了吗？**
- 环境里有没有会**静默改变实体状态**的系统服务（同步盘、存储优化、虚拟文件系统、缓存 / 占位层）？你的关键路径在它们管辖内吗？
- 这次运行会触到哪些**资源上限**（磁盘 / 文件描述符 / 内存 / 配额 / API 额度）？触到时是崩溃、静默降级还是数据污染？

> 示例（泛化，非范围边界；项目真实事故往此加）：① 云同步盘的"存储优化"把冷文件驱逐成占位符，版本控制或打包工具读到空壳后静默挂起 / 损坏；② 文件描述符上限（如 1024）在满载中途耗尽，run 跑到一半报错作废。

### 问 2 · 表征-实体对齐——所有「已决 / 已选定 / 已修复」，能逐条指认它在被测产物里的落点吗？

- 逐项核对：**跑的那个基线版本里的实际组件（方案、变体、配置、参数）== 裁决选定项**？默认继承基座不算"已接入"。
- 从共同基座派生的**所有分支 / 副本**，基座级变更都传导到了吗？（改 A 忘 B 最易发生）
- 同名不同层的组件（两层用同一个名字时），裁决点名的是**哪一层**？
- 找不到实体落点的"已决" = 纸面态，**不得上机**。

> 示例（泛化，非范围边界；项目真实事故往此加）：① 某优化"已裁定采用"但整链跑的仍是默认实现（默认继承 ≠ 已接入）；② 改了 A 分支、忘了从同一基座派生的 B 分支。

### 问 3 · 装置忠实性——测量装置自己被验证过吗？

- 新链路先用 **mock / 已知理论值对象**跑通并核对吻合了吗？（测量系统要先于被测对象验收）
- 装置承担了**不属于它的成本**吗（把被测对象的安全 / 正确性要求错安到装置上 = 伪瓶颈）？又省略了**必须的忠实度**吗？装置只需"够忠实"，但要说得出忠实在哪。
- 装置自身的**常量 / 上界 / 简化**（超时钳位、缓冲上限、开环假设），会混进被测数据、被读成被测对象的属性吗？

> 示例（泛化，非范围边界；项目真实事故往此加）：① 压测工具自己的直方图上限（如 60s 钳位）被误当成被测对象的超时哨兵；② 探针 / 胶水代码里的 bug 差点冤枉一个成熟依赖库——装置和胶水常是最大误差源。

### 问 4 · 判据完备性——判据能在过程中自动判死吗？

- 主判据挂**熔断**了吗：预热后武装、设灾难阈（远离临界带）、持续一个稳定窗越阈即停或转诊断态？
- 判据被**污染 / 干扰**时能自动识别吗（稳态判据、隔离靠机制不靠意愿）？污染数据作废留痕，绝不进判定。
- 有**实时视图**盯"判据是否已灾难性决出"，而不只盯"进程还活着 / 还在推进"吗？

> 示例（泛化，非范围边界；项目真实事故往此加）：① 首窗数据已越死线（如绝大多数请求超时）却仍空跑满数十分钟才浮出结论——缺熔断；② 抢核 / 资源争用污染了这一轮，稳态判据自动判无效并重跑（判据自愈的正面样子）。

### 问 5 · 无人值守韧性——中断、挂起、静默失败时会发生什么？

- 状态**落盘**了吗？断点续跑路径演练过吗？（超过十分钟的工作按"必被打断"设计）
- "进程活着但不推进""任务返回了但没做完"能被发现吗？**巡检兜底循环**（查磁盘状态 / 进程存活时长，不等通知）设了吗？
- 你依赖的每个"会通知我"的机制，失效时兜底是什么？（通知、返回值、时间戳都不可信，只有磁盘内容是事实）

> 示例（泛化，非范围边界；项目真实事故往此加）：① 派出去的执行 agent"启动即返回"、其实没干活，只有查磁盘产物才发现；② 一次配额 / 额度中断同时打断一批并行任务——凡状态落盘的都能恢复，凡只在对话里的全丢。

## 填写说明

- 若项目有对应机检脚本（如环境预检），可先跑收窄，但**过脚本 ≠ 本问完成**——脚本只查已知实例，未知形态靠开放式问题人工扫描。
- 各问示例区随事故一般化闭环增长：新事故先问"哪个既有问题本应逼出它"，有就往那一问加实例，没有才提请新增问题——防止本单退化成越来越长的已知坑清单。
- 反模式：① 把示例当成清单边界，只查示例列出的坑；② run 中途才想起判据没定、边跑边补。
