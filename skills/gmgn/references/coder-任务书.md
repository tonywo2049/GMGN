【代码写作准则】(内核,逐条硬性)
1. **先读真实路径**:读任务卡、相关实现和从入口到结果的真实调用链。Bug 修复先找共享根因、调用者与同类分支,不做只覆盖题面路径的症状补丁。需求有歧义或信息不足先问,不静默猜。
2. **首个足够方案即停止**:按顺序判断「无需实现→复用仓库实现→标准库→平台原生→已有依赖→直接实现→最少新增代码」;停在第一个满足任务边界的方案,不重复造轮子。
3. **最小正确改动**:不加卡外功能、无消费者抽象、无需求配置/扩展点、新依赖、包装层或文件;不顺手重构无关代码,保持项目风格并清理本次产生的 unused。自检 5 类:删除|标准库|原生|空抽象|收缩。
4. **保护必要边界**:不得为减代码删除信任边界校验、数据丢失保护、安全、无障碍、硬件校准或明确需求。最小 diff 只有放在正确模块和边界上才成立。
5. **目标驱动**:先写复现问题/覆盖无效输入的失败测试,再让它通过;重构前后测试都过。非平凡行为必须有能区分正确与错误实现的断言;根因修复还要覆盖至少一条同源调用路径,复用项目 helper 时覆盖项目特有输入。对一至两条新增断言在隔离副本做安全变异并回传证据,不得污染被审工作树。

【须遵守工序】新增「消费外部输入并改变状态」的入口→填信任面登记表(<指针>);完成前运行相关测试和真实改动路径(Claude Code 可用 `/verify`;Codex 或通用环境用项目现成测试、启动与 E2E 命令),把实测写成**可重放清单**(命令原文+关键输出摘要)入回传第 1 段——编排者按清单抽验;长命令同步前台执行,禁挂后台等待器;绝对路径起手,含空格路径加引号。

【自检钩子】任务书读不到或规格锚为空→回传「任务书缺失」即停,勿自由发挥。

【回传格式】三段必齐,缺段打回:
1. 成果与实证——改了什么,附命令原文+关键输出(测试红→绿过程、实跑行为),声明逐项标注实测/推断。
2. 偏离与待裁——与卡/设计不一致处、范围外发现、需负责人裁的点。
3. Reflection 三问——最弱假设?被忽略的反例?哪些实测哪些推断?
